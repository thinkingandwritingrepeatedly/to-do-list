# 我的想法箱 - 设计文档

## 一、项目概述

一个轻量级的个人待办事项管理工具，采用**纯前端单页应用 + 本地 Node.js 服务**架构。数据通过 HTTP API 持久化到本地文件系统，彻底解决浏览器 Local Storage 因缓存重置导致的数据丢失问题。

## 二、功能需求

### 2.1 核心功能
| 功能 | 说明 |
|------|------|
| 快速新增 | 随时记录想法/任务，支持标题 + 分类 + 备注 |
| 状态管理 | 待办 / 已完成 / 已搁置 三种状态自由切换 |
| 编辑修改 | 支持修改标题、分类、状态、备注 |
| 删除 | 删除不再需要的条目（带二次确认） |
| 搜索 | 全文搜索标题和备注内容 |
| 筛选 | 按状态筛选（全部/待办/已完成/已搁置） |
| 排序 | 按创建时间、标题排序 |
| 分类汇总 | 自动统计各分类下的条目数量 |
| 全局统计 | 总条目数、各状态数量统计 |
| 存储提示 | 顶部显示数据存储位置、文件占用空间及最后更新时间 |
| 数据导出 | 导出全部数据为 JSON 文件下载到本地 |
| 数据导入 | 从本地 JSON 文件导入数据替换当前全部数据（带确认） |
| 连接检测 | 服务断开时自动提示并禁用操作，恢复后自动重连 |

### 2.2 数据模型

```typescript
interface Todo {
  id: number;           // 时间戳 ID
  title: string;        // 标题
  notes: string;        // 备注
  category: string;     // 分类名称
  status: 'pending' | 'completed' | 'postponed';
  createdAt: string;    // ISO 创建时间
  updatedAt: string;    // ISO 更新时间
}
```

### 2.3 默认分类
- 工作 (蓝色)
- 学习 (绿色)
- 生活 (橙色)
- 灵感 (紫色)
- 其他 (灰色)

用户可在新增/编辑时输入自定义分类，系统自动识别并添加。

## 三、技术架构

### 3.1 架构总览

```
浏览器 (index.html)  ←→  Node.js HTTP 服务 (server.js)  ←→  本地文件系统 (D:\重要数据\todos.json)
```

前端通过 Fetch API 与本地服务通信，服务端负责读写磁盘 JSON 文件。服务端内置健康检查接口，前端定时轮询检测连接状态。

### 3.2 技术栈
| 层级 | 技术 | 说明 |
|------|------|------|
| UI | HTML5 + CSS3 | 语义化标签 + Flexbox/Grid 布局 |
| 样式 | 原生 CSS（CSS Variables） | 无外部依赖，加载零延迟 |
| 逻辑 | Vanilla JavaScript (ES6+) | 无需框架，轻量高效 |
| 通信 | Fetch API (HTTP) | 浏览器与本地服务之间的数据交换 |
| 服务端 | Node.js (built-in http 模块) | 零外部依赖，提供静态文件服务 + REST API |
| 存储 | 本地文件系统 (JSON) | 持久化存储在 `D:\重要数据\todos.json` |

### 3.3 服务端 (server.js)

使用 Node.js 内置 `http` 模块，无外部依赖。

**路由设计：**

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/` | 返回 index.html 首页 |
| GET | `/api/data` | 读取 `D:\重要数据\todos.json` 返回全量数据 |
| PUT | `/api/data` | 接收请求体 JSON，覆盖写入 `D:\重要数据\todos.json` |
| GET | `/api/info` | 返回文件存储信息（大小、路径、是否存在、最后修改时间） |
| GET | `/api/health` | 健康检查接口，返回服务状态和时间戳 |

**启动时自检：**
- 自动创建 `D:\重要数据` 目录（如不存在）
- 自动创建空的 `todos.json`（如不存在）
- 自动查找并清理占用同一端口的旧进程（防止 Ctrl+C 残留）

**连接检测（新增）：**
- `Store._startHeartbeat()` 每 5 秒调用 `/api/health` 检测服务连接
- 连接断开时：显示红色横幅提示、禁用输入框和添加按钮
- 连接恢复时：自动隐藏横幅、恢复交互、Toast 提示

### 3.4 前端数据流

```
[用户操作] → Store 方法 (add/update/delete)
    ↓
[校验连接] → _checkConnected() → 未连接则拦截并提示
    ↓
[内存数据更新] (Store._data)
    ↓
[异步写入] await fetch PUT /api/data → server.js → D:\重要数据\todos.json
    ↓
[失败回滚] 写入失败 → 恢复内存数据 → 断开标记 → UI 更新
    ↓
[UI 更新] Renderer.refresh() → DOM 重绘
```

**页面加载流程：**
```
DOMContentLoaded → initApp() → await Store.init()
    ↓
fetch GET /api/data → 读取 D:\重要数据\todos.json
    ↓
数据加载完成 → updateConnectionUI() → Store._startHeartbeat()
    ↓
绑定事件 → Renderer.refresh() → 首次渲染
```

### 3.5 启动方式

```bash
# 方式一：双击
start.bat

# 方式二：命令行（可指定端口）
start.bat 4567

# 方式三：npm
npm start
```

默认端口 `3456`，服务启动后自动在 http://localhost:3456 提供访问。

## 五、安全与边界

### 5.1 数据安全
- 数据存储在 `D:\重要数据\todos.json`，不依赖浏览器存储
- 清除浏览器数据不影响已保存的数据
- JSON 文件可直接用文本编辑器打开查看

### 5.2 边界处理
- 空状态：无数据时展示引导提示
- 搜索无结果：显示友好空状态提示
- 编辑时标题为空：阻止保存并提示
- 删除操作：二次确认弹窗
- 服务启动时端口被占用：自动清理旧进程
- 服务器未启动时访问页面：Store 初始化失败提示并降级为空数据
- 服务运行中断开：连接监测机制检测到断开后，显示红色横幅、禁用操作、定时重试
- 服务恢复后自动重连：Toast 提示恢复，恢复交互
- 导入数据格式非法：文件格式校验 + JSON 解析异常提示
- 导入数据不完整：校验每条数据是否含标题字段
- 写文件失败时内存数据回滚：add/update/delete 操作在 API 失败时恢复内存状态
